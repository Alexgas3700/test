## Как я могу помочь?

### 🐛 Сообщить о баге

Если вы нашли баг:

1. Проверьте, что баг еще не был [зарегистрирован](../../issues)
2. Создайте новый issue с тегом `bug`
3. Используйте шаблон для описания бага:

```markdown
**Описание бага**
Четкое описание того, что произошло.

**Как воспроизвести**
Шаги для воспроизведения:
1. Перейти к '...'
2. Кликнуть на '...'
3. Увидеть ошибку

**Ожидаемое поведение**
Что должно было произойти.

**Скриншоты**
Если применимо, добавьте скриншоты.

**Окружение:**
- n8n версия: [например, 1.0.0]
- OS: [например, Ubuntu 22.04]
- Browser: [например, Chrome 120]
- Node.js версия: [например, 18.19.0]

**Дополнительный контекст**
Любая другая полезная информация.
```

### 💡 Предложить улучшение

Если у вас есть идея:

1. Проверьте, что идея еще не была [предложена](../../issues)
2. Создайте новый issue с тегом `enhancement`
3. Опишите:
   - Какую проблему решает ваше предложение
   - Как вы видите реализацию
   - Альтернативные варианты, которые вы рассматривали

### 📝 Улучшить документацию

Документация всегда может быть лучше:

- Исправить опечатки
- Добавить примеры
- Улучшить объяснения
- Добавить переводы

### 🎨 Добавить новый email-шаблон

Мы всегда рады новым шаблонам:

1. Создайте HTML-шаблон
2. Убедитесь, что он адаптивный
3. Протестируйте на разных почтовых клиентах
4. Добавьте в `email-templates.md`
5. Создайте Pull Request

### 🔧 Улучшить workflow

Если вы хотите улучшить workflow:

1. Опишите, что вы хотите изменить
2. Объясните, почему это улучшение
3. Создайте issue для обсуждения
4. После одобрения создайте Pull Request

## Процесс разработки

### Настройка окружения

1. **Fork репозитория**

```bash
# Кликните "Fork" на GitHub
```

2. **Клонируйте ваш fork**

```bash
git clone https://github.com/ваш-username/n8n-email-campaign.git
cd n8n-email-campaign
```

3. **Создайте ветку для изменений**

```bash
git checkout -b feature/ваша-фича
# или
git checkout -b fix/ваш-баг-фикс
```

4. **Установите n8n для тестирования**

```bash
npm install n8n -g
n8n start
```

5. **Импортируйте workflow в n8n**

- Откройте http://localhost:5678
- Import from File → выберите `email-campaign-workflow.json`

### Тестирование изменений

Перед созданием Pull Request:

1. **Протестируйте workflow**
   - Импортируйте измененный workflow
   - Выполните тестовую отправку
   - Проверьте все узлы

2. **Проверьте документацию**
   - Убедитесь, что markdown корректно отображается
   - Проверьте все ссылки
   - Исправьте опечатки

3. **Протестируйте на разных окружениях** (если применимо)
   - n8n Cloud
   - n8n Self-hosted
   - Docker

### Внесение изменений

1. **Внесите изменения**

```bash
# Редактируйте файлы
```

2. **Commit изменений**

```bash
git add .
git commit -m "feat: добавить новый шаблон письма"
```

3. **Push в ваш fork**

```bash
git push origin feature/ваша-фича
```

4. **Создайте Pull Request**

- Перейдите на GitHub
- Нажмите "New Pull Request"
- Заполните шаблон PR

## Стандарты кода

### Workflow (JSON)

- Используйте понятные имена для узлов
- Добавляйте notes к узлам для пояснения
- Группируйте связанные узлы
- Используйте consistent naming convention

### JavaScript (в Code узлах)

```javascript
// ✅ Хорошо
const emailList = items.map(item => ({
  email: item.json.Email,
  name: item.json.Name || 'Valued Customer',
  company: item.json.Company || ''
}));

// ❌ Плохо
const e = items.map(i => ({
  email: i.json.Email,
  name: i.json.Name,
  company: i.json.Company
}));
```

**Правила:**
- Используйте `const` и `let`, избегайте `var`
- Используйте понятные имена переменных
- Добавляйте комментарии для сложной логики
- Обрабатывайте edge cases
- Используйте `||` для значений по умолчанию

### HTML/CSS (в email шаблонах)

```html
<!-- ✅ Хорошо -->
<table style="width: 100%; max-width: 600px;">
  <tr>
    <td style="padding: 20px; font-family: Arial, sans-serif;">
      Контент
    </td>
  </tr>
</table>

<!-- ❌ Плохо -->
<div class="container">
  <p>Контент</p>
</div>
```

**Правила:**
- Используйте inline стили (не external CSS)
- Используйте таблицы для layout (для совместимости)
- Тестируйте на Outlook, Gmail, Apple Mail
- Добавляйте alt текст к изображениям
- Максимальная ширина 600px

### Markdown (документация)

```markdown
<!-- ✅ Хорошо -->
## Заголовок раздела

Краткое описание.

### Подзаголовок

Детальное объяснение с примерами.

<!-- ❌ Плохо -->
## заголовок
текст без структуры
```

**Правила:**
- Используйте заголовки иерархически (H1 → H2 → H3)
- Добавляйте пустые строки между секциями
- Используйте code blocks для кода
- Добавляйте эмодзи для улучшения читаемости (умеренно)
- Проверяйте ссылки

## Commit сообщения

Мы используем [Conventional Commits](https://www.conventionalcommits.org/):

### Формат

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Types

- `feat`: новая функциональность
- `fix`: исправление бага
- `docs`: изменения в документации
- `style`: форматирование, отсутствующие точки с запятой и т.д.
- `refactor`: рефакторинг кода
- `test`: добавление тестов
- `chore`: обновление build задач, настроек и т.д.

### Примеры

```bash
# Новая функциональность
git commit -m "feat(workflow): add A/B testing support"

# Исправление бага
git commit -m "fix(email): correct personalization variable"

# Документация
git commit -m "docs(readme): add installation instructions"

# Рефакторинг
git commit -m "refactor(code): simplify email generation logic"
```

### Правила

- Используйте imperative mood ("add" не "added")
- Первая строка не более 72 символов
- Добавляйте body для сложных изменений
- Ссылайтесь на issues: `Closes #123`

## Pull Request процесс

### Создание PR

1. **Заполните шаблон PR**

```markdown
## Описание
Краткое описание изменений.

## Тип изменения
- [ ] Bug fix
- [ ] New feature
- [ ] Documentation update
- [ ] Refactoring

## Как тестировалось?
Опишите, как вы тестировали изменения.

## Чек-лист
- [ ] Код следует стандартам проекта
- [ ] Добавлены комментарии в сложных местах
- [ ] Документация обновлена
- [ ] Изменения протестированы
- [ ] Commit сообщения следуют конвенции

## Скриншоты (если применимо)
```

2. **Дождитесь review**

Мейнтейнеры проверят ваш PR и могут:
- Одобрить и смержить
- Запросить изменения
- Оставить комментарии

3. **Внесите запрошенные изменения**

```bash
# Внесите изменения
git add .
git commit -m "fix: address review comments"
git push origin feature/ваша-фича
```

4. **Merge**

После одобрения ваш PR будет смержен в main ветку.

### PR Guidelines

- ✅ Один PR = одна фича/фикс
- ✅ Понятное описание изменений
- ✅ Ссылка на связанный issue
- ✅ Скриншоты для UI изменений
- ✅ Обновленная документация
- ❌ Не включайте несвязанные изменения
- ❌ Не делайте слишком большие PR
